## 1. 目标与输出

使用本手册把原始文本转换为可追踪的叙事数据，不在此阶段设计景别、机位或运镜。

输出至少包括：

```text
Source Manifest
Scene List
Entity Mentions
Dialogue Blocks
Fact Ledger
Beat Sheet
Open Questions
Conflict Report
```

执行时遵守：

- 保留原文。
- 分开原文事实、视觉化、推断和授权新增。
- 让每个 Beat 指向稳定来源。
- 让每个锁定事实可以被验证。
- 不为了方便分镜提前改写剧情。

## 2. 冻结输入

### 2.1 保存版本

为每份输入记录：

```yaml
source_document_id:
title:
source_type: screenplay | outline | dialogue | storyboard | prompt | revision_note
revision:
received_at:
language:
encoding:
content_hash:
parent_revision:
author_or_provider:
authority_rank:
```

不得覆盖旧版本。任何修订创建新 `revision`。

### 2.2 确定主版本

按以下顺序选择主事实来源：

```text
用户明确锁定的修订
→ 最新被批准的剧本
→ 原始剧本
→ 梗概或对白稿
→ 已有分镜
→ 生成提示词
```

当两个同级版本冲突时停止合并冲突字段，输出冲突报告。

### 2.3 建立内容哈希

保存全文哈希和分段哈希。使用哈希检测原稿是否在下游阶段被静默改动。

不要把哈希当作来源范围。哈希只能证明内容版本，不能说明某个字段来自哪一段。

## 3. 建立 Source Manifest

### 3.1 稳定分段

按结构划分：

- 场景标题。
- 动作段。
- 人物提示。
- 对白段。
- 括注。
- 转场。
- 注释或制作说明。

为每段创建稳定 ID：

```yaml
source_span_id:
source_document_id:
source_revision:
block_type:
start_offset:
end_offset:
text:
text_hash:
parent_scene_hint:
```

使用字符偏移或行列范围时，同时保留原文片段哈希，避免上游插入文字后范围错位。

### 3.2 不稳定格式

当输入没有剧本格式时：

1. 保留原始顺序。
2. 按语义分段。
3. 把分段标记为分析结果。
4. 不把分析分段伪装成作者原格式。

### 3.3 来源范围

让每个派生对象引用一个或多个 `source_span_id`。需要跨段推导时引用全部相关范围。

禁止使用“来自整篇剧本”替代精确来源，除非该字段确实是全局规则。

## 4. 规范化但不改写原稿

### 4.1 允许的规范化

可以：

- 统一换行和空白。
- 标记人物名变体。
- 标记场景标题。
- 识别明显的全角半角差异。
- 分离动作与对白。
- 为同一人物建立别名映射。

### 4.2 禁止的静默修改

不得：

- 修正可能改变意义的错字而不保留原文。
- 将模糊人物代词直接替换成某个角色。
- 把旁白改成人物对白。
- 把梦境、回忆或想象当成现实事件。
- 把作者解释直接写成发生过的动作。
- 把分镜中的新增信息反向写入剧本事实。

### 4.3 文本修正

发现疑似错字时记录：

```yaml
normalization_issue_id:
source_span_id:
original:
suggested:
confidence:
meaning_impact:
status: unresolved | accepted | rejected
```

在未确认前继续使用原文，并在派生字段中标记不确定。

## 5. 解析 Scene

### 5.1 识别场景边界

优先使用明确场景标题。没有标题时根据以下变化判断：

- 地点。
- 时间。
- 内外景。
- 现实、回忆、梦境、监控或屏幕内容。
- 主要戏剧情境。

### 5.2 Scene 字段

记录：

```yaml
scene_id:
source_refs:
slugline_original:
slugline_normalized:
location_text:
interior_exterior:
time_of_day:
story_time:
scene_entry_state:
scene_exit_state:
characters_mentioned:
characters_present:
unresolved_presence:
```

区分“被提到”和“实际在场”。

### 5.3 场景连续性

如果同一地点在不同时间再次出现，创建新 Scene，但可以复用相同 `location_id`。

如果电话两端同时发生行动，按叙事和画面需求建立两个 Scene 或一个交叉场景组，不要假设它们处于同一物理空间。

## 6. 抽取实体与对白

### 6.1 人物

抽取：

- 显式姓名。
- 别名和称谓。
- 首次出场。
- 在场状态。
- 说话者。
- 被提及者。
- 身份和关系声明。

遇到“经理”“王总”“父亲”可能指向同一人时建立候选映射，不要直接合并。

### 6.2 地点

抽取：

- 地点名称。
- 内外景。
- 可见空间要素。
- 出入口。
- 固定物体。
- 与其他地点的关系。

不要从“公司”自动推断豪华写字楼。

### 6.3 道具

抽取所有承担以下功能的物体：

- 证据。
- 冲突触发。
- 身份证明。
- 交易媒介。
- 动作目标。
- 连续性锚点。
- 钩子或兑现。

为每个关键道具记录首次出现、持有人、位置、状态和状态变化。

### 6.4 对白

为每段对白记录：

```yaml
dialogue_id:
source_ref:
scene_id:
speaker_id:
addressee_ids:
text:
delivery_note_source:
audibility:
story_function:
fact_ids:
```

区分：

- 同期对白。
- 画外对白。
- 旁白。
- 电话或设备声音。
- 心声或作者说明。

不要把括注中的心理解释当作对白。

### 6.5 动作

把动作拆为：

```text
执行者
→ 动作
→ 对象
→ 结果
→ 受影响状态
```

例：

```text
她 / 放下 / 录音笔 / 录音笔位于桌面中央 / 证据公开出现
```

不要只记录“她反击了”。

## 7. 建立事实台账

### 7.1 事实类型

至少区分：

- 人物身份。
- 人物关系。
- 时间。
- 地点。
- 事件。
- 事件结果。
- 物品归属。
- 物品状态。
- 秘密和知情范围。
- 世界规则。
- 对白明确声明。
- 作者明确说明。
- 用户锁定要求。

### 7.2 Fact 字段

```yaml
fact_id:
fact_type:
canonical_value:
source_refs:
source_authority:
locked:
allowed_transformations:
valid_from_beat:
valid_until_beat:
known_by_character_ids:
confidence:
status: asserted | inferred | disputed | superseded
contradiction_ids:
```

### 7.3 锁定规则

默认锁定：

- 人物身份。
- 核心人物关系。
- 死亡或生存状态。
- 关键证据。
- 事件结果。
- 改变因果的时间顺序。
- 用户锁定的钩子和结局。

不要锁定纯视觉候选，例如“用红色文件夹承载合同”，除非原文明确写明红色。

### 7.4 事实与声明

区分“人物说了什么”和“客观事实是什么”。

例：

```text
经理说：“她伪造了合同。”
```

正确记录：

- `FACT-A`：经理公开声称她伪造合同。
- 不自动创建“她确实伪造合同”的客观事实。

### 7.5 事实冲突

发现冲突时：

1. 保留各自来源。
2. 标记 `disputed`。
3. 判断冲突是否为剧情设计、角色撒谎或版本错误。
4. 不自动选择更戏剧化的版本。
5. 将无法解释的冲突路由到来源确认。

## 8. 拆分 Beat

### 8.1 Beat 判断问题

对每段内容依次问：

1. 谁正在做什么？
2. 该人物想得到什么？
3. 什么阻碍了行动？
4. 什么信息发生变化？
5. 什么状态发生变化？
6. 这个变化由什么引起？
7. 下一段行动为什么会发生？

有明确变化时创建 Beat。

### 8.2 Beat 字段

记录：

```yaml
beat_id:
scene_id:
source_refs:
fact_ids:
start_state:
active_character_ids:
character_goal:
obstacle:
visible_or_audible_event:
information_before:
information_after:
result:
end_state:
causal_predecessor_ids:
causal_successor_ids:
must_preserve:
optional_visualization:
hook_role:
transformation_type:
field_provenance:
```

### 8.3 Beat 粒度

粒度过粗的迹象：

- 一个 Beat 内出现多个独立结果。
- 人物目标中途改变。
- 新证据出现后仍与出现前混在一起。
- 权力关系变化两次以上。
- 无法用一句“发生了什么变化”概括。

粒度过细的迹象：

- 每个手势都被拆成 Beat。
- 每句对白都单独成 Beat。
- 一个连续行动被无意义切碎。
- 拆分后没有新的信息、目标或状态变化。

### 8.4 多对多映射

允许：

- 一个原文范围产生多个 Beat。
- 一个 Beat 跨越多个原文范围。
- 一个 Shot 覆盖多个紧密相关 Beat。
- 一个 Beat 由多个 Shot 覆盖。

不要强制一段一 Beat 或一 Beat 一 Shot。

### 8.5 因果链

为关键 Beat 建立：

```text
原因 Beat
→ 行动 Beat
→ 结果 Beat
→ 后续反应 Beat
```

检查每个重大结果是否有前因，每个关键行动是否有动机或触发。

缺少前因时标记缺口，不擅自补剧情。

## 9. 处理心理信息与作者解释

### 9.1 分类

把不可直接拍摄的内容分为：

- 内心决定。
- 情绪总结。
- 动机解释。
- 作者评价。
- 抽象氛围。
- 关系判断。
- “不是……而是……”式解释。

### 9.2 保留叙事信息

不要简单删除心理句。先判断它承载什么：

- 人物决定。
- 关系变化。
- 信息变化。
- 情绪结果。
- 未来行动动机。

把信息保存在 Beat 或事实台账中，再交给视觉转译阶段寻找画面证据。

### 9.3 转译候选

在本阶段只记录候选方向：

```text
动作
道具变化
人物距离
表情变化
沉默或声音
对白
环境反应
```

不要在剧本解析阶段直接冻结镜头。

### 9.4 可保留的情绪词

情绪词不是自动错误。保留作为表演目标，但必须与可执行动作分开：

```yaml
performance_intent: 克制愤怒
visible_evidence_candidate: 她收紧下颌，拇指反复压住打火机盖
```

## 10. 处理梗概、对白稿与残缺输入

### 10.1 梗概

对每个事件标记：

```text
明确事实
缺失行动方式
缺失地点
缺失时间
缺失参与人物
缺失结果
```

只补足让明确事实可拍的最小条件。

### 10.2 对白稿

从对白提取：

- 说话者目标。
- 信息给予与隐瞒。
- 冲突升级。
- 必须看见的反应。
- 对话前后状态。

不要把所有对白内容当作客观事实。角色可能撒谎、误解或试探。

### 10.3 缺少人物名

使用稳定占位符：

```text
CHAR-UNKNOWN-01
CHAR-UNKNOWN-02
```

不要根据性别、职业或说话方式擅自命名。

### 10.4 缺少地点

使用 `LOC-UNKNOWN`，并记录该 Scene 对空间的最低要求，例如：

```text
至少需要一张可放置合同的桌面和一扇可供人物进入的门。
```

不要把最低空间要求升级为剧情事实。

### 10.5 缺少结局

保持 Beat 为未完成状态：

```yaml
result: unresolved
```

不要为了钩子自动补一个反转。

## 11. 处理已有分镜和提示词

### 11.1 已有分镜

反向提取：

- 声称的 Beat。
- 新增或遗漏的事实。
- 主体、动作和结果。
- 镜头内隐含剪辑。
- 与原稿冲突的对白或道具。

把分镜中的内容按来源分为：

```text
原稿支持
视觉化
推断
未经授权新增
```

### 11.2 Seedance 提示词

反向提取：

- 人物身份描述。
- 场景。
- 起始状态。
- 动作链。
- 摄影机。
- 声音。
- 尾帧。
- 负面约束。

不要把“电影感、压迫感、宿命感”解析为具体事实。

提示词中出现原稿没有的人物、道具或结果时标记潜在剧情漂移。

## 12. 验证剧情覆盖与因果

### 12.1 覆盖检查

对每个锁定 Fact 检查：

- 是否由至少一个 Beat 承载。
- 是否在需要出现的 Scene 中出现。
- 是否被后续输出错误删除或反转。

对每个 must-preserve Beat 检查：

- 是否有来源。
- 是否有明确变化。
- 是否有开始和结束状态。
- 是否连接到因果链。

### 12.2 漏剧情检查

重点检查：

- 关键证据首次出现。
- 人物第一次得知秘密。
- 决定发生的触发点。
- 冲突结果。
- 关系变化。
- 时间跳跃。
- 结局或跨集悬念。

### 12.3 重复检查

合并只重复同一信息、同一结果和同一功能的 Beat。

不要仅因文本相似合并。两个人分别得知同一秘密可能是两个不同 Beat。

### 12.4 因果检查

阻断以下情况：

- 结果先于原因且没有重排授权。
- 人物使用尚未获得的信息。
- 道具在首次出现前被使用。
- 人物在未进入场景时参与行动。
- 死亡或离场人物无解释地重新出现。

## 13. 输出契约

### 13.1 Source Manifest

至少输出：

```yaml
documents:
source_spans:
revision_graph:
primary_source_id:
authority_rules:
content_hashes:
```

### 13.2 Scene List

至少输出：

```yaml
scenes:
  - scene_id:
    source_refs:
    location_text:
    story_time:
    characters_present:
    entry_state:
    exit_state:
```

### 13.3 Fact Ledger

至少输出：

```yaml
facts:
contradictions:
unknowns:
locked_fact_ids:
```

### 13.4 Beat Sheet

至少输出：

```yaml
beats:
causal_edges:
must_preserve_beat_ids:
optional_visualization_notes:
unresolved_gaps:
```

### 13.5 分析报告

明确列出：

- 使用的主版本。
- 未解决冲突。
- 推断内容。
- 缺失信息。
- 不能在当前权限下处理的内容。
- 需要下游视觉化的抽象信息。

## 14. 正反例

### 14.1 事实与声明

原文：

```text
经理指着她说：“合同是你伪造的。”
```

正确：

```text
Beat：经理公开指控她伪造合同。
Fact：经理作出该指控。
未建立：她确实伪造合同。
```

错误：

```text
Fact：女主伪造合同。
```

### 14.2 Beat 拆分

原文：

```text
她没有解释，只把录音笔放到桌上。会议室安静下来，经理伸向文件的手停住。
```

合理拆分：

```text
BEAT-01：她公开录音证据。
BEAT-02：众人意识到信息局势变化。
BEAT-03：经理停止继续控制文件。
```

错误拆分：

```text
她抬手 / 她伸手 / 她碰到桌子 / 她放下录音笔
```

这些动作本身没有独立叙事变化。

### 14.3 心理信息

原文：

```text
她终于不再忍耐。
```

正确分析：

```text
叙事信息：她作出停止服从的决定。
视觉化状态：待下游处理。
```

错误分析：

```text
事实：她把辞职信推到经理面前。
```

原文未提供辞职信；该动作只能是候选视觉化，不是原文事实。

### 14.4 梗概

原文：

```text
男主发现合伙人背叛了他。
```

正确：

- 锁定“发现背叛”这一事件。
- 将背叛方式、证据形式和地点标记为未知。

错误：

- 自动补成合伙人转走公司五百万元。

## 15. 错误路由

| 错误 | 严重级别 | 返回阶段 | 处理 |
|---|---|---|---|
| 原稿未冻结 | blocker | 输入锁定 | 保存版本和哈希后重做分析 |
| 来源范围不稳定 | error | Source Manifest | 重建稳定 span |
| 两个主版本冲突 | blocker | 版本确认 | 保留冲突，指定权威版本 |
| 人物别名错误合并 | error | 实体解析 | 拆分实体并重算引用 |
| 角色声明被当成事实 | blocker | Fact Ledger | 改为声明事实 |
| 锁定事实无来源 | blocker | Fact Ledger | 补来源或取消锁定 |
| Beat 没有状态变化 | warning | Beat 拆解 | 合并或说明其铺垫功能 |
| Beat 粒度过粗 | error | Beat 拆解 | 按独立变化拆分 |
| 关键因果缺失 | blocker | Beat/来源确认 | 找回原文或标记剧情缺口 |
| 推断伪装成原文 | blocker | Provenance | 改为 `inferred` |
| 授权新增无授权 ID | blocker | 权限控制 | 删除或取得授权 |
| 抽象句被直接删除 | error | 抽象信息分类 | 恢复其叙事信息并待视觉化 |
| 分镜新增事实反写原稿 | blocker | 反向分析 | 从事实台账移除 |
| 缺少地点或人物 | warning | Open Questions | 使用未知占位符，不编造 |

## 16. 完成门禁

只有满足以下条件才把结果交给 Story Bible 和镜头阶段：

- 原始输入和所有修订已冻结。
- 已确定主版本或标明版本冲突。
- 每个 Scene 有稳定来源。
- 每个关键人物、地点和道具有稳定 ID。
- 每段关键对白有说话者或未知占位符。
- 每个锁定 Fact 有明确来源。
- 角色声明与客观事实已分开。
- 每个 must-preserve Beat 有开始状态、变化和结束状态。
- 关键 Beat 已建立因果关系。
- 心理信息已保留其叙事含义。
- 所有推断和授权新增已明确标记。
- 没有在剧本分析阶段提前冻结摄影方案。
