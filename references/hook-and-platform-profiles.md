# 钩子契约与平台构图 Profile

先确认原稿已经承诺什么，再决定如何前置、延迟和兑现。把每个钩子写成可验证的 `Hook Contract`，不得用空泛旁白、无因反转或未授权新事实制造“抓人”。在镜头设计前加载版本化 Platform Profile，把画幅、安全区、字幕区、界面遮挡区和首屏读取条件写入调度、Coverage、Camera Setup、Shot 与 Hook Contract。

## 适用范围

在以下情况加载本手册：

- 用户要求钩子、爽点、反转、留存或段尾悬念。
- 输入是竖屏短剧、剧情短视频、信息流剧情广告或横屏微短片。
- 需要判断开场多久出现异常、冲突或信息缺口。
- 需要为字幕、标题、头像、评论、点赞、进度条或平台控件预留画面区域。
- 需要把横屏剧本重新设计为竖屏分镜。
- 需要检查钩子是否兑现，或是否被合法延迟到下一集。

不要把“用户想要抓人”自动解释为允许改写剧情。先读取 `modification_policy` 和授权记录。

## 权限边界

### 忠实转换 `faithful`

只允许：

- 从原稿已有事件中选择更早可见的证据。
- 把已有结果、异常、危险或身份线索前置为冷开场。
- 调整画面揭示顺序，但不改变因果。
- 用更清楚的动作、道具、声音和构图强化原有冲突。

禁止：

- 新增背叛、死亡、怀孕、犯罪、血缘、巨额财富、警察到场等事实。
- 让人物提前知道原稿中尚未知晓的信息。
- 改变事件结果或角色立场。

### 视觉增强 `visual`

允许增强证据可读性、动作结果、空间压力和声音提示。不得新增剧情事实。

### 节奏优化 `pacing`

允许在不改变核心因果的前提下前置揭示、压缩重复、重排反应镜头和设置合法延期。所有重排必须记录来源与差异。

### 剧情优化 `story`

只有存在明确授权时，才允许新增钩子、误导、反转或冲突。每个新增事实必须标记：

```yaml
transformation_type: authorized_addition
authorization_id:
```

没有授权 ID 时，停止写入正式分镜。

## 钩子的操作定义

把钩子定义为：

> 在明确时间窗口内，用可见或可听证据提出一个与人物目标、风险或信息变化有关的问题，并在当前交付范围内兑现、合法延期或经授权保留开放。

一个有效钩子至少包含：

```text
证据
+ 观众问题
+ 代价
+ 延迟结构
+ 兑现状态
```

以下不构成完整钩子：

- “事情没那么简单。”
- “三天后，她将震惊所有人。”
- 只有强音乐，没有事件变化。
- 随机耳光、摔杯或尖叫，但不影响人物目标。
- 画面看见一个神秘物体，之后不再出现。

## Hook Contract

### 使用既有字段

每个正式钩子至少记录：

```yaml
hook_id:
job_id:
episode_id:
hook_type:
promise_beat_ids:
promise_shot_ids:
viewer_question:
stakes:
visible_or_audible_evidence:
delay_beat_ids:
payoff_beat_ids:
payoff_shot_ids:
payoff_type:
payoff_evidence:
status:
deferred_to_episode:
next_episode_bridge_beat_ids:
authorized_change:
authorization_id:
platform_profile_id:
genre_profile_id:
target_window:
field_provenance:
```

不要只写“钩子：女主身份反转”。必须绑定承诺和兑现的 Beat、Shot 与证据。

### viewer_question

把问题写成观众可理解、后续可回答的具体问题：

```text
好：录音笔里是谁的声音？
好：合同上的授权日期为什么晚于签署日期？
好：门外的人是否拿着女主丢失的工牌？

坏：接下来会发生什么？
坏：她的命运将走向何方？
```

### stakes

说明若问题不解决，人物具体会失去什么：

- 工作。
- 证据。
- 时间。
- 人身安全。
- 关系。
- 公开信誉。
- 行动机会。

不要只写“后果严重”。

### visible_or_audible_evidence

记录摄像机或麦克风能捕捉的证据：

```text
手机屏幕倒计时从 00:05 跳到 00:04。
门把手从外侧缓慢压下。
录音中出现经理说出的合同编号。
```

不要记录作者解释：

```text
她意识到危险正在逼近。
```

## 承诺、延期与兑现

### 承诺

承诺必须发生在 `promise_beat_ids` 和 `promise_shot_ids` 中，并在目标平台窗口内可读。

承诺可以是：

- 一个异常。
- 一个未完成动作。
- 一个具体危险。
- 一个身份证据。
- 一个违反规则的事实。
- 一个迫近时限。
- 一个公开冲突。

承诺不能依赖导演备注才能理解。

### 延期

只在以下条件下延期：

1. 中间 Beat 持续增加相关证据、阻碍或代价。
2. 延期没有用无关支线拖延。
3. 观众问题保持一致，或明确升级为更具体的问题。
4. 延期长度符合 Platform 与 Genre Profile。
5. 当前交付结束时能指向明确 `deferred_to_episode`。

把每个延期 Beat 标记为以下功能之一：

```text
evidence_added
obstacle_added
stakes_raised
false_answer_retracted
action_interrupted
perspective_shifted
```

不要使用与钩子无关的争吵填充延期。

### 兑现

兑现必须回答或实质改变原问题：

- `answer`：直接回答。
- `partial_answer`：回答一部分，并产生更具体的新问题。
- `reversal`：原先合理推断被新证据推翻。
- `escalation`：问题的危险或范围被明确扩大。
- `identity_reveal`：身份被可验证揭示。
- `evidence_reveal`：证据内容被清楚读取。
- `consequence`：承诺的行动产生明确结果。
- `deferred`：合法指向后续集。
- `open_ending`：只有获得开放结局授权时使用。

`payoff_evidence` 必须是可见或可听结果。

### 关闭条件

当前交付结束时，每个 Hook Contract 必须满足以下一种：

```text
status: paid_off
```

且存在有效 `payoff_beat_ids` 与兑现证据；

或：

```text
status: deferred_to_episode
```

且 `deferred_to_episode` 与 `next_episode_bridge_beat_ids` 有效；

或：

```text
status: intentionally_unresolved
```

且有明确授权。

否则标记 `blocked`。

## 钩子类型

### 结果前置 `result_first`

先显示已有结果，再回到因果过程。

适合：

- 婚礼中断。
- 合同被撕。
- 人物被赶出现场。
- 关键道具已经损坏。

检查：原稿必须真的包含该结果；回切过程不得改变时间线。

### 视觉异常 `visual_anomaly`

让正常环境中出现具体异常：

- 已锁门的门把手转动。
- 所有人手机同时收到同一张照片。
- 空房间的打印机吐出新文件。

异常必须在后续解释或合法延期。

### 身份反差 `identity_contrast`

让人物当前待遇与可验证身份信息发生冲突。

不要仅靠昂贵服装暗示身份；使用证件、授权、他人行为或系统信息。

### 迫近时限 `deadline`

使用真实存在的倒计时、截止动作或即将失效的机会。

时限必须影响行动，不要添加无因倒计时。

### 公开冲突 `public_conflict`

在他人面前发生可见选择、拒绝、指控、驱逐或夺权。

不要为了开场刺激擅自增加耳光或暴力。

### 逼近危险 `approaching_danger`

让观众先看到人物尚未处理的具体危险。

危险来源必须属于故事世界，并受安全与合规规则约束。

### 规则破坏 `rule_break`

先建立规则，再显示规则被违反：

- 已注销账号重新登录。
- 已封存文件出现新签字。
- 宣布死亡的人发来通话。

若规则未建立，观众无法判断异常。

### 关键证据 `critical_evidence`

展示能够改变局势的录音、合同、监控、转账、物证或消息。

保证内容在平台安全区内可读，并有足够停留时间。

### 未完成行动 `unfinished_action`

在动作结果出现前切断：

- 门即将打开。
- 文件即将上传。
- 手即将按下删除键。

下一段必须继续该行动或明确解释中断原因。

### 戏剧性反讽 `dramatic_irony`

观众知道人物尚不知道的信息。

严格维护知识状态；不要让人物无依据突然作出正确判断。

## 钩子设计流程

按以下顺序执行：

1. 读取事实台账、Beat Sheet、修改权限、平台和题材 Profile。
2. 标出原稿已经存在的异常、风险、证据、时限和未完成行动。
3. 选择与主角当前目标直接相关的一项。
4. 写出具体 `viewer_question`。
5. 写出具体 `stakes`。
6. 选择最少的可见或可听证据。
7. 指定承诺 Beat 与 Shot。
8. 指定每个延期 Beat 的功能。
9. 指定兑现 Beat、Shot、类型和证据。
10. 检查目标窗口和平台安全区。
11. 检查是否新增事实或提前泄密。
12. 运行 Hook Contract 闭环校验。

若原稿没有足够钩子，在未授权时输出“可选创意建议”，不要写入正式分镜。

## 钩子强度评分

评分只用于排序和审查，不把主观分数当作事实。

### 可计算子项

每项 0–5：

```yaml
evidence_clarity:
question_specificity:
stakes_specificity:
goal_relevance:
novel_information:
time_to_comprehension:
payoff_traceability:
platform_legibility:
```

建议计算：

```text
hook_score =
证据清晰度 × 0.20
+ 问题具体度 × 0.15
+ 代价具体度 × 0.15
+ 人物目标相关度 × 0.15
+ 新信息量 × 0.10
+ 理解速度 × 0.10
+ 兑现可追踪性 × 0.10
+ 平台可读性 × 0.05
```

将结果归一化为 0–100。

### 否决条件

即使总分高，也不得通过：

- 新增事实无授权。
- 承诺无兑现或合法延期。
- 证据不在画面或声音中。
- 关键文字不可读。
- 钩子与人物目标无关。
- 依赖危险、违法或平台禁限内容但没有处理方案。

## 平台 Profile

### 不写死平台规则

平台 UI、画幅和发布要求会变化。把快速变化信息放入版本化 Profile：

```yaml
platform_profile_id:
profile_version:
target_platform:
content_surface:
verified_at:
evidence_source_ids:
aspect_ratios:
default_aspect_ratio:
resolution_profiles:
safe_zones:
subtitle_zones:
ui_occlusion_zones:
title_zones:
first_frame_requirements:
hook_windows:
duration_ranges:
audio_defaults:
caption_defaults:
stale_after_days:
status:
```

不得在核心规则中宣称某平台永久固定某个像素安全区。

### 使用归一化坐标

优先使用从 0 到 1 的归一化画面坐标：

```yaml
zone_id:
x_min:
y_min:
x_max:
y_max:
applies_to:
priority: hard | soft
```

坐标原点和方向必须在 Profile 中声明：

```yaml
coordinate_origin: top_left
x_direction: right
y_direction: down
```

再根据目标分辨率换算像素，避免 1080×1920 数值被错误用于其他分辨率。

### 记录 Profile 新鲜度

执行：

1. 检查 `verified_at`。
2. 检查当前日期是否超过 `stale_after_days`。
3. 若过期，标记 `stale`。
4. 不把过期 Profile 的 UI 遮挡区作为已验证事实。
5. 允许用户提供当前平台截图作为运行时覆盖。

## 竖屏安全区

### 在镜头设计前介入

按以下顺序执行：

```text
平台与画幅
→ 安全区和 UI 遮挡区
→ Blocking
→ Coverage
→ Camera Setup
→ Final Shot
```

不要在横屏镜头全部完成后再简单裁成 9:16。

### 定义区域

至少区分：

- **核心叙事安全区**：脸、手、关键道具和动作结果必须清楚可见。
- **字幕安全区**：对白字幕、说明字幕可稳定放置。
- **标题区**：开场标题或章节文字区域。
- **UI 遮挡区**：头像、互动按钮、描述、评论提示、进度条等可能覆盖区域。
- **软边缘区**：允许背景或非关键肢体进入，但不放关键证据。

关键证据不得与硬 UI 遮挡区相交。

### 可计算检查

对关键对象记录边界框：

```yaml
subject_or_prop_id:
frame_range:
bbox_normalized:
importance: critical | supporting | decorative
required_zone_id:
occlusion_tolerance:
subtitle_conflict_allowed:
```

计算：

```text
critical_overlap_ratio =
关键对象边界框与 UI 遮挡区交集面积
/ 关键对象边界框面积
```

阈值由 Platform Profile 指定。超过硬阈值时阻断。

### 字幕与画面冲突

当字幕覆盖手部、手机、合同或嘴部时：

1. 优先重新构图。
2. 调整人物站位或道具高度。
3. 调整字幕区。
4. 必要时拆出独立 Insert。

不要通过缩小关键道具来给字幕让位。

### 输出干净版需求

需要多平台适配时，记录：

- 无字幕干净画面。
- 字幕版。
- 平台 UI 预览版。
- 横竖屏替代构图。

不要把烧录字幕版本作为唯一母版。

## 竖屏调度与 Coverage

### 单人镜头

- 利用上半身、手部和道具形成纵向动作。
- 保留头顶与字幕空间。
- 避免关键手部动作掉出底部 UI 区。
- 不要让人物长期贴住画面边缘。

### 双人镜头

优先使用：

- 前后景层次。
- 上下层次。
- 一坐一站。
- 门框或桌面形成纵深。
- 过肩和单人反应覆盖。

不要把两人机械挤在左右边缘，导致脸和视线都进入 UI 区。

### 多人镜头

执行：

1. 确定当前主导主体。
2. 把其余人物组织成前、中、后景。
3. 只为会改变局势的反应创建独立 Coverage。
4. 必要时拆镜，不让所有人物同时争抢竖屏宽度。

### 关键道具

合同、手机、证件、录音笔等必须：

- 位于核心叙事安全区。
- 有足够画面占比。
- 有足够读取时长。
- 避开字幕和 UI。
- 在前后 Shot 中保持方向与手部连续性。

### 横屏转竖屏

不要直接裁切。重新检查：

- 双人空间关系是否丢失。
- 出入画方向是否仍清楚。
- 视线目标是否被裁掉。
- 关键道具是否离开安全区。
- Master 是否仍能建立空间。
- Coverage 是否需要补单人、过肩或 Insert。

## 平台内容类型 Profile

以下为内容类型基线，不代表具体平台永久规则。实际时间窗口必须由版本化 Platform Profile 提供。

### 竖屏连续短剧

目标：

- 快速建立人物目标、关系或危险。
- 保持每个钩子与主线因果相连。
- 在段尾留下明确行动、问题或后果。

优先：

- 可读身份或证据。
- 公开选择。
- 未完成行动。
- 反应和道具 Insert。

避免：

- 每集重复同一误会。
- 只延迟不增加证据。
- 所有角色轮流大段解释。

### 剧情类短视频

目标：

- 在单条视频内形成完整微型因果。
- 让钩子获得兑现，不把所有答案推到下一条。

优先：

- 一个核心问题。
- 一次主要转折。
- 一个可见结果。

### 信息流剧情广告

目标：

- 明确产品、服务或行动与人物问题的关系。
- 避免用虚假倒计时、虚假身份或无法证明的效果做钩子。

必须读取品牌、广告和平台合规 Profile。不要在本手册中自行断言营销承诺合法。

### 横屏微短片

允许更长的空间建立与关系镜头，但仍应尽早出现人物目标或变化。不要机械套用竖屏高切换率。

### 悬疑连续内容

允许问题分层延期，但每个阶段必须新增证据或收紧可能性。不得无限堆谜。

### 喜剧段子

钩子通常建立规则、误解或视觉预期；兑现通过反转、重复升级或具体动作完成。笑点后保留可读反应，不要用解释台词复述笑点。

## 首屏、前段与段尾

### 首屏

首屏必须能够快速回答至少两项：

- 现在发生了什么异常或变化？
- 谁受到影响？
- 观众应该看哪里？
- 哪个物体或动作最重要？

不要要求首帧解释完整背景。

### 前段

在 Platform Profile 的前段窗口内：

1. 建立具体问题。
2. 建立人物目标或代价。
3. 避免连续抽象旁白。
4. 确保字幕和关键画面不冲突。
5. 让第一个信息变化发生在可计算窗口内。

不要在核心规则中固定“前 2 秒必须反转”。不同内容类型使用不同窗口。

### 段尾

段尾可使用：

- 行动即将完成。
- 新证据刚出现。
- 人物作出不可撤销选择。
- 危险进入画面。
- 已有问题获得部分回答并升级。

段尾不能只靠：

- 突然黑屏。
- “未完待续”字幕。
- 无关尖叫。
- 没有来源的神秘来电。

## 正反例

### 未授权钩子

```text
坏：
原稿只是职场争执，开场新增警察把经理带走。

好：
开场先拍原稿已有的录音笔红灯；经理在画外说“你没有证据”。
```

```text
坏：
为了爽点，让女主突然宣布自己是董事长女儿。

好：
若原稿已有董事会授权，开场先显示授权文件上的姓名和会议室门禁放行结果。
```

### 承诺与兑现

```text
坏：
开场门把手转动，之后再也没有人进门，也没有解释。

好：
Hook Contract 记录“门外是谁”；第三个 Beat 显示来人手里拿着女主遗失的工牌，完成身份线索兑现。
```

```text
坏：
手机倒计时只是装饰，归零后没有结果。

好：
倒计时对应文件撤回窗口；归零时上传按钮变灰，人物失去撤回机会。
```

### 延期

```text
坏：
连续三场无关争吵，仍不回答录音内容。

好：
第一次延期增加录音文件损坏；第二次延期找到云端备份；兑现时播放出具体合同编号。
```

### 竖屏安全区

```text
坏：
手机屏幕放在右侧互动按钮下方，发送时间被遮住。

好：
手机移到核心叙事安全区；发送时间保持足够画面占比，并预览字幕和 UI 叠加。
```

```text
坏：
横屏双人远景直接裁成竖屏，只剩两个人各半张脸。

好：
重新调度为前后景双人关系镜头，并补关键人物单人反应。
```

### 首屏

```text
坏：
黑屏旁白：“她的人生从今天改变。”

好：
门禁屏显示“权限已撤销”，她的工牌贴上读卡器后亮红灯。
```

### 段尾

```text
坏：
人物正常说完一句话，突然加重低音并黑屏。

好：
人物按下播放键；录音里刚出现经理的声音，画面切黑。下一集桥接 Beat 继续播放内容。
```

### 钩子评分

```text
坏：
因为“很刺激”，直接给 95 分。

好：
分别记录证据清晰度、问题具体度、代价、目标相关度、理解速度和兑现追踪；同时检查否决条件。
```

## 硬失败

出现以下任一项，必须退回：

- 钩子新增事实但没有授权 ID。
- 钩子改变人物知识状态、身份、关系或事件结果。
- `viewer_question` 无法被后续回答。
- 承诺没有可见或可听证据。
- 当前范围结束时既未兑现，也未合法延期或授权开放。
- 延期 Beat 与原问题无关。
- 兑现证据无法在画面或声音中读取。
- `critical_evidence` 位于硬 UI 遮挡区。
- Platform Profile 缺少版本、核验日期或来源状态。
- 使用过期 Profile 却宣称平台规则已验证。
- 安全区坐标没有坐标系说明。
- 横屏直接裁竖屏导致关键人物、视线、动作或道具丢失。
- 关键字幕与关键画面重叠且没有解决方案。
- 信息流广告用虚假时限、虚假身份或未验证效果制造钩子。
- 段尾只用黑屏、音乐或“未完待续”替代剧情状态变化。

## 软警告

以下情况允许保留，但必须说明：

- 首屏同时出现两个以上竞争性问题。
- 钩子理解依赖小字或复杂屏幕信息。
- 连续多个 Hook Contract 共用同一兑现 Beat。
- `partial_answer` 没有生成更具体的新问题。
- 依赖误导，但没有足够公平线索。
- 同一异常被反复延期，新增证据不足。
- 竖屏双人镜头持续较长，个别反应无法独立调整。
- UI 遮挡数据来自用户截图而非官方说明。
- Profile 接近过期。
- 同一项目需要多个平台版本，但只设计了一套构图。
- 字幕位置需要频繁跳动以躲避关键画面。
- 钩子评分高，但与人物情感或目标关系弱。

## 错误路由

| 问题 | 返回阶段 | 处理 |
|---|---|---|
| 新增事实无授权 | 权限与事实台账 | 删除或取得授权 |
| 钩子问题不具体 | Beat 与钩子设计 | 重写 viewer_question |
| 代价空泛 | 人物目标分析 | 写明具体失去内容 |
| 无兑现 | Hook Contract | 补兑现、合法延期或阻断 |
| 延期拖沓 | 节奏设计 | 删除无关 Beat，增加相关证据 |
| 证据不可拍 | 视觉转译 | 改成动作、道具或声音证据 |
| 文字不可读 | Coverage 与 Shot | 增加 Insert 或延长读取时间 |
| 竖屏构图冲突 | Blocking | 重排人物和道具位置 |
| UI 遮挡 | Platform Profile / Camera Setup | 更新区域或构图 |
| Profile 过期 | 来源核验 | 更新证据和版本 |
| 段尾空悬念 | Beat 设计 | 绑定未完成行动或新后果 |
