## 1. 适用范围

本手册用于已经存在初版剧本分析、Story Bible、分镜或生成任务后，继续执行以下工作：

- 根据用户反馈继续开发后续场景、下一集或未完成段落。
- 修复“前后接不上”“动作跳了”“镜头有点散”“节奏断了”等非结构化反馈。
- 在不重做全部产物的前提下，局部优化 Shot、Generation Task、Asset 或 Timeline。
- 将实拍、生成或剪辑装配暴露的新约束回写到后续方案。
- 对多轮修改保留基线、差异、验证证据和回滚点。

本手册不扩大修改权限。先按 [工作流模式与权限控制](workflow-modes.md) 锁定 `faithful`、`visual`、`pacing` 或 `story`，再决定允许修复到哪一层。

## 2. 核心原则

1. **先诊断，后修改。** 不把“不连贯”直接解释为需要重写剧情。
2. **修上游责任层。** Beat 错误回到 Beat，调度错误回到 Blocking，不用下游提示词掩盖上游问题。
3. **最小影响面。** 只修改造成问题的最小对象集合，并明确哪些下游产物因此失效。
4. **保留稳定 ID。** 未改变语义和边界的对象保留 ID；拆分、合并或替换时记录映射。
5. **新旧版本可比较。** 每轮都保存基线、修复前后差异、验证结果和未解决项。
6. **生成结果不覆盖事实。** Asset 可以暴露新制作约束，不能反向改变原稿事实或人物身份。
7. **权限不足即停。** 需要新增剧情、改变因果或接受已知缺陷时，提交建议并等待授权。

## 3. 冻结迭代基线

开始修改前冻结最近一个已接受版本。至少记录：

```yaml
iteration_id:
baseline:
  source_version:
  story_bible_version:
  scene_and_beat_version:
  storyboard_version:
  continuity_ledger_version:
  generation_task_version:
  timeline_version:
  quality_report_version:
feedback:
  feedback_id:
  original_text:
  affected_scene_ids: []
  affected_shot_ids: []
  expected_result:
revision_permission:
production_mode:
```

以上是迭代记录示例，不是新的正式 Schema。正式机器产物继续使用 [Schema 索引](schema-index.md) 中的现有对象；修改操作优先写入 `RepairPatch`，运行状态优先写入 `RunSnapshot`。

如果用户只说“这些部分不太连贯”，先完成可继续进行的诊断，再列出定位问题所需的最少信息，例如：

- 指出最明显的 Scene、Shot 或时间段。
- 说明是剧情、画面、动作、声音、节奏还是人物表现接不上。
- 提供当前采用的剧本和分镜版本。
- 说明希望保持原顺序，还是允许局部调序。

不得因反馈不够精确而从头重写全部分镜。

## 4. 诊断“不连贯”

### 4.1 六层诊断

按从上游到下游的顺序检查：

| 层级 | 常见表现 | 返回阶段 | 主要检查对象 |
|---|---|---|---|
| 剧情与信息 | 因果缺口、人物提前知道信息、目标突然改变 | Source / Fact / Story Bible | Fact Ledger、知识状态、人物目标 |
| Scene 与 Beat | 场景入口无承接、情绪或权力变化没有触发事件 | Scene / Beat | Scene 状态快照、Beat 进入与退出状态 |
| 调度与空间 | 人物瞬移、越轴、视线错误、道具换手无过程 | Blocking / Continuity | Mark、路线、轴线、动作相位、连续性账本 |
| Shot 与剪辑 | 镜头缺少切点理由、反应缺失、动作匹配失败 | Coverage / Shot | Coverage、Shot Intent、首尾状态、剪辑闭环 |
| 声音与节奏 | 台词抢时、环境声断裂、停顿或声画桥缺失 | Timing / Audio | Audio Event、关键路径时长、Room Tone、声画桥 |
| 生成与装配 | 角色外观漂移、首尾帧不接、Asset 与 Shot 不一致 | Task / Asset / Timeline | Task 约束、Attempt、Asset 可用区间、Clip |

上游层一旦发现阻断问题，暂停对其下游的润色。先修复根因，再重新生成或验证受影响对象。

### 4.2 建立证据窗口

每个问题至少保留一个“前—中—后”证据窗口：

```text
前：问题对象之前最近一个已接受状态
中：用户指出或验证器发现的问题对象
后：问题对象之后第一个依赖其结果的对象
```

对相邻 Shot，逐项比较：

- 上一镜 `end_state` 与下一镜 `start_state`。
- 主体位置、面向、视线、姿势和动作相位。
- 道具所有者、所在手、开关和损耗状态。
- 服装、伤势、污渍、天气、灯光和时间。
- 人物已知信息、当前目标和表演强度。
- 环境声、对白尾音、音乐和声画桥。
- 切镜前的信息结果与切镜后的观众问题。

将结论标记为：

- `continuous`：状态可直接承接。
- `explained_change`：变化有时间、动作、声音或 Scene 转换依据。
- `repair_required`：存在无来源跳变或剪辑断点。
- `blocked`：需要新事实、权限或用户确认。

这些标记用于审查记录，不替代现有 Schema 枚举。

### 4.3 区分连续性问题与创作偏好

以下通常属于可验证的不连贯：

- 上一镜动作未完成，下一镜已进入不兼容相位。
- 人物、道具、视线、画面方向或声音无来源跳变。
- 关键问题尚未建立，镜头却直接给出反应或答案。
- Scene 出口状态无法承接下一个相关 Scene 的入口。
- Shot、Generation Task 和最终 Asset 表达不同事件。

以下通常属于偏好，需要进一步说明目标：

- “不够高级”“不够电影感”“节奏不喜欢”。
- 希望增加新的冲突、反转、人物关系或剧情结果。
- 希望改变题材风格、叙事顺序或结局。

把偏好转成可检查目标，例如“缩短信息重复”“在揭示证据前保留一个可读反应”“保持 A 左、B 右的画面方向”。

## 5. 执行最小修复闭环

按以下顺序执行一轮迭代：

1. 冻结基线与用户原始反馈。
2. 建立证据窗口并定位最早出现错误的责任层。
3. 列出必须保持不变的事实、镜头和已批准资产。
4. 生成最小 RepairPatch；不要直接覆盖基线。
5. 检查补丁是否超出 `revision_permission`。
6. 从责任层开始修改，并向下游传播失效范围。
7. 更新相邻对象的进入与退出状态，不只改用户指出的单个镜头。
8. 重新运行受影响的验证器和聚合验证。
9. 用同一证据窗口比较修复前后结果。
10. 输出已解决项、未解决项、回滚点和下一轮建议。

### 5.1 失效传播

修改后按依赖方向标记和重建：

```text
Source / Fact
  → Story Bible
  → Scene / Beat
  → Blocking / Coverage / Camera Setup
  → Shot
  → Generation Task
  → Attempt / Asset
  → Timeline Clip
  → Quality Report / Release Checklist
```

示例：

- 只调整 Shot 时长：重验 Timing、Audio、Task 和 Timeline，不必重写 Story Bible。
- 修改人物入口位置：更新 Blocking、受影响 Shot、Task 和 Continuity Ledger。
- 修改 Beat 结果：其后依赖该结果的 Shot、Task、Asset 和 Clip 全部标记为待复核。
- 替换 Asset：不自动修改 Shot；先验证新 Asset 是否满足原 Shot 与连续性目标。

### 5.2 最小补丁优先级

按以下顺序选择能解决问题的最小方法：

1. 补充缺失状态或来源引用。
2. 调整动作相位、时长、声音或剪辑点。
3. 增加原剧情已有信息所需的反应或证据 Coverage。
4. 拆分过载 Shot 或合并无独立信息结果的重复 Shot。
5. 局部重排原文已有信息。
6. 切换真人、Seedance 或混合制作方式。
7. 申请 `story` 权限后才允许新增或改变剧情。

## 6. 继续开发新场景或新集

后续开发不是重新初始化项目。开始新内容前继承最近一个已接受状态：

- Story Bible 中的人物身份、关系、目标、知识状态和行为锚点。
- 上一个相关 Scene 的出口快照。
- 未兑现的 Hook Contract、承诺、风险和延期状态。
- 仍在场或将再次出现的道具、伤势、服装和环境状态。
- 已建立的空间方向、视觉参考和声音锚点。
- 已批准的制作方式、模型档案和权利限制。

执行：

1. 先写新 Scene 的入口状态，再拆新 Beat。
2. 将入口与上一个相关 Scene 的出口逐项比较。
3. 对时间、地点或人物组合变化建立可见或可听的重建依据。
4. 新增的第一个 Shot 必须继承入口状态，不得从默认空状态开始。
5. 只有确实缺少承接信息时才增加 Bridge Beat 或 Bridge Shot。
6. Bridge 不得引入原稿不存在的关键事实；需要新增时请求授权。
7. 新内容完成后反查旧 Hook、人物目标和道具是否得到合法延续或兑现。

读取 [Story Bible](story-bible.md) 的场景状态快照与版本管理规则，以及 [人物调度与连续性](blocking-and-continuity.md) 的镜头间状态交接规则。

## 7. 使用三镜窗口优化分镜

不要孤立优化单镜。对每个目标镜头 `S(n)`，同时检查 `S(n-1)` 和 `S(n+1)`。

### 7.1 进入镜头

确认 `S(n-1) → S(n)`：

- 切镜理由明确：新信息、反应、证据、动作相位或注意力中心发生变化。
- 画面方向、视线和动作相位可承接。
- 声音没有无依据中断。
- 观众知道此刻应该看什么、听什么。

### 7.2 镜头内部

确认 `S(n)`：

- 只有一个主导景别、主要行动和叙事目的。
- 开始状态、可见动作和结束状态完整。
- 动作、对白、反应、摄影机和声音能在时长内完成。
- 关键动作没有落入遮挡区、字幕区或画幅外。
- 真人或生成方式与镜头复杂度匹配。

### 7.3 退出镜头

确认 `S(n) → S(n+1)`：

- 本镜的信息结果已被读到，或明确设计为跨镜延续。
- 结束状态为下一镜提供可执行的起点。
- 反应、证据或声音闭环没有被提前切断。
- 若切到新 Scene，已提供时间、地点或状态重建。

Coverage 与切镜理由见 [剪辑与 Coverage](editing-and-coverage.md)；对白、动作和声音的关键路径见 [计时与声音](timing-and-audio.md)。

完成拆镜或合镜后，按 Beat 重新组装镜头组，检查“准备 → 行动 → 结果 → 反应”的因果链。没有 Beat 或 Coverage 作用的孤立 Shot 应删除、合并或说明保留理由；重复传递同一信息的 Shot 应压缩，除非它承担新的反应、证据或节奏功能。

## 8. 同步生成任务、素材与时间线

当 Shot 改动时：

- Shot 边界变化：拆分或重建对应 Generation Task，不复用旧任务假装已同步。
- 首尾状态变化：更新参考素材、首帧状态、动作时间线、尾帧目标和连续性验收条件。
- 时长变化：重新检查动作、对白、声音、生成时长和 Timeline Clip。
- 制作方式变化：记录切换理由、交接资产和回退条件。
- Asset 已生成：将旧 Asset 标记为候选、部分可用或失效，不静默覆盖。

真实 Asset 暴露新约束时：

1. 判断它是制作约束还是剧情事实冲突。
2. 制作约束可写回 Continuity Ledger、Task 或后续 Shot。
3. 与锁定事实冲突的 Asset 必须淘汰、局部修复或重新生成。
4. 保留当前最佳 Task、Asset、失败 Attempt 和回滚点。

读取 [生成风险、Attempt 评审与分支修复](generation-risk-and-repair.md)；不要为了保留一次生成结果而改写后续全部剧情。

## 9. 管理反馈轮次与停止条件

默认最多执行三轮最小补丁。每轮只处理已确认的问题集合：

```text
第 1 轮：修复 Blocker 和根因层错误
第 2 轮：修复受影响的下游一致性与可执行性
第 3 轮：处理仍影响观看或交付的 Error，并完成复核
```

满足任一条件即停止自动迭代并请求人工决定：

- 同一阻断问题连续两轮没有改善。
- 修复一个问题会引入更严重的事实、连续性或权利冲突。
- 需要改变剧情事实、人物身份、核心因果或结局。
- 需要接受含已知阻断缺陷的 Asset。
- 继续尝试将超过预算、时限或允许的 Attempt 数。
- 用户反馈目标彼此冲突，无法在当前权限内同时满足。

不要用无限“优化”掩盖缺少明确目标或授权。

## 10. 交付格式

每轮至少交付以下内容：

### 10.1 反馈诊断表

| 反馈 ID | 证据窗口 | 问题层级 | 根因 | 严重度 | 修复权限 | 责任阶段 |
|---|---|---|---|---|---|---|
| F001 | S010–S012 | Shot 与剪辑 | 反应 Coverage 缺失 | error | faithful | Coverage / Shot |

### 10.2 修复与影响表

| 修复对象 | 修复前 | 最小补丁 | 受影响下游 | 验证 | 状态 |
|---|---|---|---|---|---|
| S011 | 证据后直接切场 | 增加原 Beat 已包含的反应 Shot | Task、Timeline | Coverage、Timing、Continuity | resolved |

### 10.3 迭代摘要

```yaml
iteration_id:
baseline_version:
resolved_feedback_ids: []
unresolved_feedback_ids: []
changed_artifact_ids: []
invalidated_artifact_ids: []
validation_results: []
authorization_requests: []
rollback_target:
next_recommendation:
```

该摘要是人类可读的交付组织方式。正式字段和枚举仍以相应 Schema 为准。

## 11. 示例

用户反馈：

> 女主看到证据后马上切到下一个场景，感觉不连贯。

错误处理：

> 新增一段女主回忆童年、发誓复仇的剧情。

该处理新增了动机和剧情事实，超出 `faithful` 权限。

正确诊断：

```text
前：S020 展示证据，结束状态为女主看清内容。
中：缺少证据被理解后的可读结果。
后：S021 已进入下一 Scene，女主开始行动。
根因：must-have reaction Coverage 缺失，不是人物动机缺失。
```

最小修复：

```text
在 S020 后增加一个独立反应 Shot：
女主停止翻页，手指压住证据中的关键位置，抬眼看向门外。
```

前提是“她理解证据并准备行动”已经存在于原 Beat。修复后：

- 更新 Shot ID 映射和差异记录。
- 继承证据、手部、视线和声音状态。
- 重新验证 Coverage、Timing、Continuity、Task 和 Timeline。
- 不新增童年、复仇或其他原稿不存在的事实。

## 12. 完成门禁

只有全部满足时，才能声明本轮反馈已解决：

- [ ] 用户原始反馈已保留并关联到明确证据窗口。
- [ ] 已定位最早责任层，没有用下游润色掩盖上游错误。
- [ ] 修复未超出修改权限，新增事实均有授权。
- [ ] 修改对象、保留对象和失效对象范围明确。
- [ ] 相邻 Shot 的进入与退出状态可承接。
- [ ] 新 Scene 入口能承接上一个相关 Scene 的出口。
- [ ] Shot、Generation Task、Asset 和 Timeline 重新对齐。
- [ ] 受影响验证器已重新运行，Blocker 与 Error 为 0。
- [ ] 修复前后差异、未解决项和回滚点已交付。
- [ ] 缺少真实生成、走戏或剪辑证据时，只声明“方案通过”。
